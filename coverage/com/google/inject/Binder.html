30    /**
31    * Collects configuration information (primarily <i>bindings</i>) which will be
32    * used to create an {@link Injector}. Guice provides this object to your
33    * application's {@link Module} implementors so they may each contribute
34    * their own bindings and other registrations.
35    *
36    * <h3>The Guice Binding EDSL</h3>
37    *
38    * Guice uses an <i>embedded domain-specific language</i>, or EDSL, to help you
39    * create bindings simply and readably. This approach is great for overall
40    * usability, but it does come with a small cost: <b>it is difficult to
41    * learn how to use the Binding EDSL in the usual way -- by reading
42    * method-level javadocs</b>. Instead, you should consult this series of
43    * examples below. To save space, these examples omit the opening
44    * {@code binder.}, just as you will if your module extends
45    * {@link AbstractModule}.
46    *
47    * <pre>
48    * bind(ServiceImpl.class);</pre>
49    *
50    * This statement does essentially nothing; it "binds the {@code ServiceImpl}
51    * class to itself" and does not change Guice's default behavior. You may still
52    * want to use this if you prefer your {@link Module} class to serve as an
53    * explicit <i>manifest</i> for the services it provides. Also, in rare cases,
54    * Guice may be unable to validate a binding at injector creation time unless it
55    * is given explicitly.
56    *
57    * <pre>
58    * bind(Service.class).to(ServiceImpl.class);</pre>
59    *
60    * Specifies that a request for a {@code Service} instance with no binding
61    * annotations should be treated as if it were a request for a
62    * {@code ServiceImpl} instance. This <i>overrides</i> the function of any
63    * {@link ImplementedBy @ImplementedBy} or {@link ProvidedBy @ProvidedBy}
64    * annotations found on {@code Service}, since Guice will have already
65    * "moved on" to {@code ServiceImpl} before it reaches the point when it starts
66    * looking for these annotations.
67    *
68    * <pre>
69    * bind(Service.class).toProvider(ServiceProvider.class);</pre>
70    *
71    * In this example, {@code ServiceProvider} must extend or implement
72    * {@code Provider<Service>}. This binding specifies that Guice should resolve
73    * an unannotated injection request for {@code Service} by first resolving an
74    * instance of {@code ServiceProvider} in the regular way, then calling
75    * {@link Provider#get get()} on the resulting Provider instance to obtain the
76    * {@code Service} instance.
77    *
78    * <p>The {@link Provider} you use here does not have to be a "factory"; that
79    * is, a provider which always <i>creates</i> each instance it provides.
80    * However, this is generally a good practice to follow. You can then use
81    * Guice's concept of {@link Scope scopes} to guide when creation should happen
82    * -- "letting Guice work for you".
83    *
84    * <pre>
85    * bind(Service.class).annotatedWith(Red.class).to(ServiceImpl.class);</pre>
86    *
87    * Like the previous example, but only applies to injection requests that use
88    * the binding annotation {@code @Red}. If your module also includes bindings
89    * for particular <i>values</i> of the {@code @Red} annotation (see below),
90    * then this binding will serve as a "catch-all" for any values of {@code @Red}
91    * that have no exact match in the bindings.
92    *
93    * <pre>
94    * bind(ServiceImpl.class).in(Singleton.class);
95    * // or, alternatively
96    * bind(ServiceImpl.class).in(Scopes.SINGLETON);</pre>
97    *
98    * Either of these statements places the {@code ServiceImpl} class into
99    * singleton scope. Guice will create only one instance of {@code ServiceImpl}
100    * and will reuse it for all injection requests of this type. Note that it is
101    * still possible to bind another instance of {@code ServiceImpl} if the second
102    * binding is qualified by an annotation as in the previous example. Guice is
103    * not overly concerned with <i>preventing</i> you from creating multiple
104    * instances of your "singletons", only with <i>enabling</i> your application to
105    * share only one instance if that's all you tell Guice you need.
106    *
107    * <p><b>Note:</b> a scope specified in this way <i>overrides</i> any scope that
108    * was specified with an annotation on the {@code ServiceImpl} class.
109    *
110    * <p>Besides {@link Singleton}/{@link Scopes#SINGLETON}, there are
111    * servlet-specific scopes available in
112    * {@code com.google.inject.servlet.ServletScopes}, and your Modules can
113    * contribute their own custom scopes for use here as well.
114    *
115    * <pre>
116    * bind(new TypeLiteral&lt;PaymentService&lt;CreditCard>>() {})
117    * .to(CreditCardPaymentService.class);</pre>
118    *
119    * This admittedly odd construct is the way to bind a parameterized type. It
120    * tells Guice how to honor an injection request for an element of type
121    * {@code PaymentService<CreditCard>}. The class
122    * {@code CreditCardPaymentService} must implement the
123    * {@code PaymentService<CreditCard>} interface. Guice cannot currently bind or
124    * inject a generic type, such as {@code Set<E>}; all type parameters must be
125    * fully specified.
126    *
127    * <pre>
128    * bind(Service.class).toInstance(new ServiceImpl());
129    * // or, alternatively
130    * bind(Service.class).toInstance(SomeLegacyRegistry.getService());</pre>
131    *
132    * In this example, your module itself, <i>not Guice</i>, takes responsibility
133    * for obtaining a {@code ServiceImpl} instance, then asks Guice to always use
134    * this single instance to fulfill all {@code Service} injection requests. When
135    * the {@link Injector} is first created, it will automatically perform field
136    * and method injection for this instance, but any injectable constructor on
137    * {@code ServiceImpl} is simply ignored. Note that using this approach results
138    * in "eager loading" behavior that you can't control.
139    *
140    * <pre>
141    * bindConstant().annotatedWith(ServerHost.class).to(args[0]);</pre>
142    *
143    * Sets up a constant binding. Constant bindings are typeless in Guice; you
144    * can provide the values in a variety of types and the values can be injected
145    * in a variety of types; Guice performs the standard type conversions for you
146    * behind the scenes. Because of this "typelessness", constant injections must
147    * always be annotated.
148    *
149    * <pre>
150    * {@literal @}Color("red") Color red; // A member variable (field)
151    * . . .
152    * red = MyModule.class.getField("red").getAnnotation(Color.class);
153    * bind(Service.class).annotatedWith(red).to(RedService.class);</pre>
154    *
155    * If your binding annotation has parameters you can apply different bindings to
156    * different specific values of your annotation. Getting your hands on the
157    * right instance of the annotation is a bit of a pain -- one approach, shown
158    * above, is to apply a prototype annotation to a field in your module class, so
159    * that you can read this annotation instance and give it to Guice.
160    *
161    * <pre>
162    * bind(Service.class)
163    * .annotatedWith(Names.named("blue"))
164    * .to(BlueService.class);</pre>
165    *
166    * Differentiating by names is a common enough use case that we provided a
167    * standard annotation, {@link com.google.inject.name.Named @Named}. Because of
168    * Guice's library support, binding by name is quite easier than in the
169    * arbitrary binding annotation case we just saw. However, remember that these
170    * names will live in a single flat namespace with all the other names used in
171    * your application.
172    *
173    * <p>The above list of examples is far from exhaustive. If you can think of
174    * how the concepts of one example might coexist with the concepts from another,
175    * you can most likely weave the two together. If the two concepts make no
176    * sense with each other, you most likely won't be able to do it. In a few
177    * cases Guice will let something bogus slip by, and will then inform you of
178    * the problems at runtime, as soon as you try to create your Injector.
179    *
180    * <p>The other methods of Binder such as {@link #bindScope},
181    * {@link #bindInterceptor}, {@link #install}, {@link #requestStaticInjection},
182    * {@link #addError} and {@link #currentStage} are not part of the Binding EDSL;
183    * you can learn how to use these in the usual way, from the method
184    * documentation.
185    *
186    * @author crazybob@google.com (Bob Lee)
187    */
